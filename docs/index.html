Health Articles — Import Process & Reference

How the Kaiser Permanente Northern California health & wellness articles were brought into this Edge Delivery Services project: the article template, the page-metadata-driven tag/share UI, and the query index that powers article listings. This work is the article-rendering foundation for Scenario 1 — Health & Wellness Landing Page. Captures the work done on 9 June 2026.

Project overview

This project recreates Kaiser Permanente's Northern California health & wellness articles as Edge Delivery Services pages. It is built on the Author Kit boilerplate, where the runtime loads through scripts/ak.js and scripts/scripts.js rather than the classic aem.js path.

The 9 June work delivered a reusable article template (CSS + JS) that matches the look of a production KP health article, plus a query index so those articles can be listed and filtered.

Where this fits in the POC. This is part of Scenario 1 — Health & Wellness Landing Page (reference: /northern-california/health-wellness). The healtharticles index targets exactly that path, and the article template renders the individual articles the landing page links to. Still outstanding for Scenario 1: the Content Hub API call to fetch content fragments and the dynamic related-articles rendering via content selectors.

Article template

The template lives in templates/article/ and activates when a page declares template: article in its metadata. The expected authored DOM:

body.article-template main .section.tags.share # the article body .default-content h1 # title p # byline "by Author | Date" p > picture # hero image p, h2, ul, ol … # body content .section.footnotes # footnotes / sources

article.js

On load, decorateArticle() finds the .section.tags.share section and appends two pieces to its content:

FunctionWhat it does
getTags()Reads repeated <meta property="article:tag">, falling back to a single comma-separated <meta name="tags">.
buildPills()Renders a labelled row of pill links; each pill links to /pages/search?query={tag}.
buildShare()Renders Email / Facebook / Twitter / Pinterest share links from the current URL + title (only when the section also has the share class).

Query index

The healtharticles index in helix-query.yaml crawls published article pages and produces a JSON feed that a listing/filter block can consume.

healtharticles: include: - /northern-california/health-wellness/** target: /northern-california/health-wellness/query-index.json properties: title, description, date, lastModified, featured, author, image, content, tags
PropertySource
titlemeta[property="og:title"]
descriptionmeta[name="description"]
datemeta[name="publication-date"] (MM-DD-YYYY)
authormeta[name="author"]
imagemeta[property="og:image"] (path only)
tagsmeta[property="article:tag"] (repeated)
featuredmeta[name="template"] ⚠︎ see open questions
contentFirst 200 chars of main > div text

Open questions & issues

Things worth confirming before this is considered done — these surfaced while reviewing the 9 June commits.

1. What sets the body.article-template class? The CSS is scoped to it and scripts.js loads the JS from the template meta, but it's worth confirming ak.js actually emits a <template>-template body class for the value article.
2. Tag pills link to /pages/search?query=… — that's the KP production search path. Does an equivalent search route exist on this EDS site? If not, every pill is a dead link.
3. The featured index field maps to meta[name="template"]. That stores the template name ("article") under a field called featured, which is semantically confusing. Is that intentional (a flag for "this is a templated article"), or a copy-paste from the blog index that should point at a real featured meta?
4. Where do the article documents live? No article content (HTML/MD) is committed to the repo — only the template and index. Confirm the actual pages are authored in DA at /northern-california/health-wellness/… so the index has something to crawl, and that a listing block exists to consume query-index.json.

Developer notes

JSTemplate JS loader

ak.js loads template CSS only. scripts.js dynamically imports templates/<name>/<name>.js after loadArea() and calls its default export.

JSMetadata-driven UI

Tags and template come from page <meta> — no block authoring needed. Authors only add a tags, share section style.

CSSBody-class scoping

All template CSS is scoped to .article-template so it never leaks into other page types in the shared codebase.

CFGQuery index

helix-query.yaml is auto-generated metadata. After editing, the index rebuilds on publish; reindex via the admin API if needed.
