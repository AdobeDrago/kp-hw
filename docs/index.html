Storybook

Storybook hosts the KP “vessel” design-system reference stories and an EDS-block harness. It is the design source of truth that drives how blocks are built and styled: each block is built to reproduce its reference story, and a visual-parity diff keeps the two from drifting.

Reference story → Shared renderer → Block init + CSS slice → Parity diff

What it is

Reference stories live under stories/<pattern>/ (cleaned vessel patterns). The EDS-block harness lives under stories/_eds/ and renders the real block from authored content via its own init(), so a block can be reviewed and pixel-compared side by side with its reference.

Running it

npm run storybook          # dev server at http://localhost:6006
npm run build-storybook    # production build (also a compile check)

npm run tokens:ds          # regenerate the --ds-* token layer from the foundation
npm run parity             # visual-parity gate (needs Storybook running)

Storybook → block recipe

The repeatable steps, as proven on card and notification.

Prereq — the pattern’s reference story exists under stories/<pattern>/ with its foundation.css vendored, and npm run tokens:ds has been run.

  1. Shared renderer Create blocks/<name>/<name>-dom.js: a pure function that takes content slots and returns the vessel DOM element (.ds-<name>). Put any behavior (e.g. dismiss) here. Take resolved asset URLs as parameters — don’t hardcode paths.
  2. Point the reference story at it In stories/<pattern>/<Name>.stories.js, import the renderer; keep ensurePatternStyles and asset resolution; pass the slots. Story and block now share one renderer and cannot drift.
  3. Block init(el) In blocks/<name>/<name>.js, read the variant from the block class, parse the authored rows into the renderer’s args, then el.replaceChildren(render(args)). Resolve block-owned assets with new URL('./icons/x.svg', import.meta.url) (works in both Vite and EDS).
  4. Ship assets Copy any icons/images the component needs into blocks/<name>/.
  5. Generate the CSS slice 
    node scripts/slice-block-css.mjs \
  --src stories/<pattern>/foundation.css \
  --match .ds-<name>[,.primitive] --root .<name> \
  --out blocks/<name>/<name>.css
    --match can list several roots (e.g. .ds-card,.button) when the component leans on a shared primitive.
  6. EDS harness stories Add stories/_eds/<Name>.eds.stories.js: one story per reference variant, authored as a table that yields the same content, with the same wrapping width / section framing as the reference. Use edsBlock({ name, rows, variants, section }).
  7. Add parity pairs Extend PAIRS in scripts/visual-parity.mjs (selector = .ds-<name>).
  8. Verify Run npm run storybook, then npm run parity; iterate the slice/CSS until the pairs pass. Lint with npx eslint and npx stylelint.

Fast feedback loop

For steps 5–8: regenerate the slice → reload the EDS story → npm run parity → open the diff image at test/visual/__output__/<pair>/diff.png to see exactly what differs, adjust, and repeat.

Gotcha — after editing story files, the dev server’s _eds index sometimes goes stale and a story 404s. Restart with rm -rf node_modules/.cache/storybook then npm run storybook.
