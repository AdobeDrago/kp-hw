Kaiser Permanente · Proof of Concept

Health & Wellness POC

A proof of concept validating Kaiser Permanente's migration from AEM Managed Services to Edge Delivery Services, and confirming EDS can power dynamic, backend-driven experiences — not just static content. Four reference scenarios from the KP Health & Wellness site drive the work. This guide tracks the architecture, engineering approach, and progress to date.

Project overview

This is a proof of concept validating Kaiser Permanente's migration from AEM Managed Services (AEM MS) to Edge Delivery Services (EDS), and proving out EDS support for dynamic, backend-driven experiences. Four reference scenarios from the production KP Health & Wellness site anchor the effort, backed by a shared design system and engineering foundation.

Purpose

  • Validate migration from AEM Managed Services to Edge Delivery Services.
  • Validate support for dynamic, backend-driven experiences.

Architecture requirements

  • One Repo Per Project architecture.
  • A referenceable, immutable Global Library — shared blocks, CSS, and design patterns aligned to the KP Design System.
  • Project repository contains blocks, pages, services, configuration, and tests.

Core engineering requirements

  • Use the EDS block-based development model only.
  • Integrate backend services: Content Hub, Lucid Search, AEM Servlet APIs.
  • Support region and language switching.
  • Implement a modular service layer with error handling for all APIs.

Article template

The template lives in templates/article/ and activates when a page declares template: article in its metadata. The expected authored DOM:

body.article-template main .section.tags.share # the article body .default-content h1 # title p # byline "by Author | Date" p > picture # hero image p, h2, ul, ol … # body content .section.footnotes # footnotes / sources

article.js

On load, decorateArticle() finds the .section.tags.share section and appends two pieces to its content:

FunctionWhat it does
getTags()Reads repeated <meta property="article:tag">, falling back to a single comma-separated <meta name="tags">.
buildPills()Renders a labelled row of pill links; each pill links to /pages/search?query={tag}.
buildShare()Renders Email / Facebook / Twitter / Pinterest share links from the current URL + title (only when the section also has the share class).

Open questions & issues

Things worth confirming before this is considered done — these surfaced while reviewing the 9 June commits.

1. What sets the body.article-template class? The CSS is scoped to it and scripts.js loads the JS from the template meta, but it's worth confirming ak.js actually emits a <template>-template body class for the value article.
2. Tag pills link to /pages/search?query=… — that's the KP production search path. Does an equivalent search route exist on this EDS site? If not, every pill is a dead link.
3. The featured index field maps to meta[name="template"]. That stores the template name ("article") under a field called featured, which is semantically confusing. Is that intentional (a flag for "this is a templated article"), or a copy-paste from the blog index that should point at a real featured meta?
4. Where do the article documents live? No article content (HTML/MD) is committed to the repo — only the template and index. Confirm the actual pages are authored in DA at /northern-california/health-wellness/… so the index has something to crawl, and that a listing block exists to consume query-index.json.

Storybook Design System In progress

How the KP "vessel" design system drives building and styling EDS blocks, using Storybook as the design source of truth. The reference-story side is built; the implementation side (story → shipping block) is in progress.

The big idea — two contracts

There are two contracts in play, and the whole effort reconciles them:

  • Storybook components are driven by args/props.
  • EDS blocks are driven by authored content (a table of rows/cells that init(el) decorates into DOM).

One design (a vessel pattern) produces one block. Storybook is the contract; the block is the implementation; a visual parity diff is the enforcement — so the story and the shipping block can never silently drift apart.

End-to-end pipeline

# DESIGN CONTRACT (built) KP vessel design system ──► Storybook reference story (compiled Sass exports) render(args) + foundation.css │ # THE EDS BLOCK — one source of truth (Phase 4) shared renderer · init(el): authored table → args → DOM CSS slice + tokens · UE content model from argTypes │ # VERIFICATION visual parity diff ◄── block vs reference story │ # AUTHORING + RUNTIME author in UE ──► EDS decorates ──► published page
No drift, by construction. The story's render() moves into the block; both the story and init() import it. A DS update → re-run tokens:ds, re-slice CSS, update the story → the parity diff shows exactly what moved.

Locked decisions

DecisionChoice
Story frameworkVanilla HTML/CSS (@storybook/html-vite) — maps closely to EDS DOM
TokensNamespaced --ds-* :root layer derived from the compiled CSS catalog
Token namespace--ds-*, separate from the existing generic --color-* palette (never touch it)
DOM sourceBlock owns the renderer; the story imports it
Existing blocksExtend/replace, don't duplicate (vessel card evolves blocks/card)
CSS scopingPer block — strip the .ds-foundation wrapper and rescope to the block root

What's built vs. planned

  • Token extractor — npm run tokens:ds
    Parses foundation.css → styles/ds-tokens.css: 49 colors (with paired --ds-on-*) + a 5-step type scale.
  • EDS-block harness — stories/_eds/eds-harness.js
    edsBlock({ name, rows, variants }) mounts any real block from an authored table, in the EDS cascade, beside its reference story.
  • CSS-slice / rescope tool — scripts/slice-block-css.mjs
    Brace-aware parser keeps one component's rules, strips the theme wrapper, tokenizes color literals. Card: 14.5k lines → 291 rules.
  • Card pilot (#12) — wired end-to-end, verified visually
    card-dom.js shared renderer · card.js init() · generated card.css slice. Matches the reference story; closing tasks remain.
  • Playwright visual-parity diff (CI gate)
    Screenshot diff per variant; fails CI on divergence. Manual comparison done; automation pending.
  • Shared-primitive hoisting · asset pipeline · scale-out
    Hoist shared rules (typography, .button, icon font) to global styles/; migrate icon fonts/SVGs; remaining patterns (#13).

Run it

# Storybook dev server (reference + EDS-block harness stories) npm run storybook # http://localhost:6006 # (Re)generate the --ds-* token layer from the vessel foundation npm run tokens:ds # writes styles/ds-tokens.css # Slice one component's CSS out of a pattern foundation node scripts/slice-block-css.mjs \ --src stories/card/foundation.css --match .ds-card --root .card --out <file.css>
Risks to keep in mind: EDS pages have no .ds-foundation wrapper, so slices must be rescoped per block. The type scale is base-only (responsive bump not yet captured). Each pattern's foundation.css is ~800 KB — never ship full foundations to EDS. Icon-font/SVG assets still need an EDS migration path. Existing header/hero/footer use Lit — refactor vs. keep is a per-block call.

Developer notes

JSTemplate JS loader

ak.js loads template CSS only. scripts.js dynamically imports templates/<name>/<name>.js after loadArea() and calls its default export.

JSMetadata-driven UI

Tags and template come from page <meta> — no block authoring needed. Authors only add a tags, share section style.

CSSBody-class scoping

All template CSS is scoped to .article-template so it never leaks into other page types in the shared codebase.

CFGQuery index

helix-query.yaml is auto-generated metadata. After editing, the index rebuilds on publish; reindex via the admin API if needed.
